### 1.2 必须掌握的 5 个核心命令

**你只需要学会这5个命令，就可以应付90%的日常开发：**

```bash
# 1. 克隆仓库到本地（第一次用）
git clone https://github.com/heituanzi-jpg/ranphicat-website.git

# 2. 查看当前状态（每次改代码前看看）
git status

# 3. 添加修改的文件（告诉Git你要提交哪些）
git add .

# 4. 提交并写备注（必须写清楚改了什么）
git commit -m "fix: 修复首页导航栏样式问题"

# 5. 推送到GitHub（同步到云端）
git push
```

**进阶命令（需要时再学）：**

```bash
# 看提交历史（理解项目演变）
git log

# 创建新分支（开发新功能时用）
git checkout -b feature/new-button

# 合并分支（功能开发完后用）
git merge feature/new-button
```

#### 笔记模板

```markdown
---
tags: [css, layout]
created: 2024-01-15
related: [[flex-direction]], [[justify-content]]
---

# Flexbox布局

## 一句话解释
Flexbox是一种一维布局模型，用于在一条轴线上排列元素。

## 核心概念
- **容器（flex container）**：设置 `display: flex` 的父元素
- **项目（flex item）**：容器内的直接子元素
- **主轴（main axis）**：项目排列的方向（默认水平）
- **交叉轴（cross axis）**：垂直于主轴的方向

## 容器属性
```css
.container {
    display: flex;
    
    /* 主轴方向 */
    flex-direction: row | column;
    
    /* 主轴对齐方式 */
    justify-content: flex-start | center | flex-end | space-between;
    
    /* 交叉轴对齐方式 */
    align-items: stretch | center | flex-start | flex-end;
    
    /* 换行方式 */
    flex-wrap: nowrap | wrap;
}
```

## 项目属性
```css
.item {
    /* 放大比例 */
    flex-grow: 0;
    
    /* 缩小比例 */
    flex-shrink: 1;
    
    /* 初始大小 */
    flex-basis: auto;
    
    /* 简写 */
    flex: 1;  /* = flex-grow: 1; flex-shrink: 1; flex-basis: 0%; */
}
```

## 实战案例：导航栏
```html
<nav class="navbar">
    <div class="logo">Logo</div>
    <ul class="menu">
        <li>首页</li>
        <li>关于</li>
        <li>联系</li>
    </ul>
</nav>
```

```css
.navbar {
    display: flex;
    justify-content: space-between;  /* logo在左，菜单在右 */
    align-items: center;              /* 垂直居中 */
    padding: 0 20px;
    background: #333;
    color: #fff;
}
```

## 踩坑记录
- ❌ 误以为 align-items 是控制主轴对齐
- ✅ 正确：align-items 控制交叉轴对齐
- ❌ 忘记给容器加 display: flex
- ✅ 教训：先确认父元素有没有设置 flex

## 关联知识点
- [[grid-layout]] - 二维布局
- [[position]] - 定位布局

## 自测问题
1. 如何让三个元素平均分布在容器中？
2. align-items 和 align-content 的区别是什么？
```

## 4. 与Trae智能体协作的正确方式

### 4.1 当前问题

你说"Trae直接改代码，看不到学习过程"，这是因为协作方式不对。

### 4.2 正确的协作方式

#### ❌ 错误示范

```
"帮我把首页改成深色模式"
→ Trae直接改了，你不知道它怎么改的
```

#### ✅ 正确示范（分三步）

**第一步：让它解释要改什么**

```
"请解释一下要把首页改成深色模式，需要修改哪些地方？"
```

**第二步：让它告诉你怎么改（不要直接执行）**

```
"请告诉我具体的修改方案，包括要改哪些文件、哪些CSS属性"
```

**第三步：让它改，但要求加详细注释**

```
"请按照上面的方案修改，同时在每个改动的位置加上中文注释，
让我理解为什么要这样改"
```

---

### 4.3 具体提问模板

#### 场景1：学习某个CSS属性

```
"我想让这个div垂直居中，请告诉我用哪种CSS方法，
然后加详细注释解释为什么这样用，不要直接改代码"
```

#### 场景2：改代码前理解原理

```
"请先解释一下Flex布局的 justify-content 和 align-items 
分别控制什么，然后告诉我现在代码里哪里用到了"
```

#### 场景3：让Trae帮你写带注释的代码

```
"请帮我写一个导航栏的HTML和CSS，要求：
1. 包含3个菜单项
2. 悬停时有颜色变化
3. 每个CSS属性都要加中文注释，解释作用
4. 不要太长，我需要理解每一行"
```

#### 场景4：改完后验证理解

```
"请用简短的话告诉我，刚才的修改主要改了什么？
如果我想要另一种效果（比如改成圆形头像），应该改哪个属性？"
```

```
1. 设置 → 第三方插件 → 开启社区插件
2. 搜索 "Obsidian Git" → 安装
3. 设置 → Obsidian Git
   - Auto backup interval: 60（每60分钟自动备份）
   - Custom message: auto backup {{date}}
   - 建议开启：Show status bar

4. 每次写完笔记后
   - 按 Ctrl+P → 输入 "Obsidian Git: commit"
   - 或等自动备份
```

---

> **记住：学习编程不是看完视频就学会了，是改出第一个bug并解决它才算开始。**
>
> 每天改5个地方，Git坚持commit，笔记自己写，Trae用来提问而不是替代你思考。
>
> **GitHub追踪进度，Obsidian深度理解，两者结合才能记得牢。**
